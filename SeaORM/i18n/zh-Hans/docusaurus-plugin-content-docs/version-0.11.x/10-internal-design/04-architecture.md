# 架构

> 让我们深入海底 🤿

<img width="100%" src="/SeaORM/img/SeaORM Architecture.svg" />

要理解 SeaORM 的架构，让我们先讨论一下 ORM 是什么。ORM 存在的目的是为数据库操作提供抽象，并隐藏实际的 SQL 查询等实现细节。

有了一个好的 ORM，你不应该需要关心 API 表面之下的内容。直到你确实需要关注。我听到你说 *"抽象泄露"*，是的，确实如此。

SeaORM 采取的方法是 **"分层抽象"**，如果你愿意的话，你可以深入到下一层。这就是为什么我们将 SeaQuery 设计为一个独立的仓库。它本身就很有用，而且有了公共的 API 表面和独立的命名空间，相比于整体式方法，创建令人困惑的内部 API 的难度要大得多。

SeaORM 的核心理念是几乎一切都是运行时可配置的。在编译时，它并不知道底层数据库是什么。

数据库无关性带来了哪些好处？例如，你可以：

1. 根据运行时配置，让你的应用在任何数据库上运行
1. 使用相同的模型并在不同的数据库之间迁移它们
1. 通过创建一个 "数据结构 crate" 来在不同的项目中共享实体，其中数据库由下游的 "行为 crates" 选择

SeaORM 的 API 并不是一个薄壳，而是由层次构成的，每一层都比上一层更少抽象。

在使用 API 时有不同阶段。

因此，有两个维度来导航 SeaORM 的代码库，**"阶段"** 和 **"抽象性"**。

首先是声明阶段。使用 `EntityTrait`、`ColumnTrait` 和 `RelationTrait` 等定义实体及其之间的关系。

第二是查询构建阶段。

最上层是 `Entity` 的 `find*`、`insert`、`update` 和 `delete` 方法，在这里你可以直观地执行基本的 CRUD 操作。

再深入一层，是 `Select`、`Insert`、`Update` 和 `Delete` 结构体，它们各自有自己的 API 用于更高级的操作。

再深入一层，是 SeaQuery 的 `SelectStatement`、`InsertStatement`、`UpdateStatement` 和 `DeleteStatement`，它们拥有丰富的 API，供你摆弄 SQL 语法树。

第三是执行阶段。一组独立的结构体，`Selector`、`Inserter`、`Updater` 和 `Deleter`，负责对数据库连接执行语句。

最后是解析阶段，当查询结果被转换成 Rust 类型并填充到结构体中。随后，如果这是一个关联查询，结构体将根据它们的关系拼接在一起。

因为只有执行和解析阶段是数据库特定的，我们有可能通过替换这些部分来使用不同的驱动程序。

我想象有一天，我们将支持多个数据库，具有不同的语法、协议和形态。
