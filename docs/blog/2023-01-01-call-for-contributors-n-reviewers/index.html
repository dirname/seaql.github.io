寻求贡献者和审阅者 📢

January 1, 2023 · 7 min read
Chris Tsang
SeaQL 创始人

我们正在寻求 SeaQL 项目的贡献者和审阅者 📢！

在过去一年中，SeaQL 的用户基础稳步增长，我们很高兴能够帮助个人和初创企业在 Rust 中构建他们的项目。然而，问题、提问和拉取请求的数量几乎达到了我们核心成员的处理能力上限。

再次感谢大家参与社区！

如果您的项目依赖于 SeaQL，并且您想帮助我们，这里有一些建议（如果您还没有，请给 我们所有的代码库 星标，并在 Twitter 上关注我们）：

  1. 财务支持。您可以在 GitHub 上赞助我们，这些资金将用于覆盖我们的开销。作为一种礼遇，我们会倾听赞助者的需求和使用案例，并不时沟通我们的组织发展情况。
  2. 代码贡献。打开一个 PR 总是受到欢迎！要开始，您可以浏览我们的 issue 跟踪器并选择一个来处理。如果您考虑开发一个重要的功能，可以先起草一个 “提案和实施计划”（PIP）。
  3. 知识贡献。知识共享有多种形式：教程、食谱、问答 和 Discord。您可以向我们的 文档 代码库提交 PR，或者在自己的平台上发布。我们将很高兴在我们的学习资源部分列出它。请关注我们的 GitHub 讨论和 Discord，帮助他人！
  4. 代码审阅。这是我们工程中的重要流程。目前，只有 3 名核心成员充当审阅者。非核心成员也可以成为审阅者，我邀请您加入我们！

现在，我想概述一下我们的审阅政策：对于成熟项目，每个合并的 PR 必须获得至少两个审阅者的批准，其中一个必须是核心成员；允许自我审阅。以下是一些示例：

  • 核心成员提交了 PR，另一名核心成员批准 ✅
  • 核心成员提交了 PR，一名审阅者批准 ✅
  • 一名审阅者提交了 PR，核心成员批准 ✅
  • 一名审阅者提交了 PR，另一名审阅者批准 ⛔
  • 一名贡献者提交了 PR，2 名核心成员批准 ✅
  • 一名贡献者提交了 PR，一名核心成员和一名审阅者批准 ✅
  • 一名贡献者提交了 PR，2 名审阅者批准 ⛔

总而言之，每个 PR 至少应该经过两双值得信赖的眼睛审查。

审查 PR 时的标准是什么？​

以下问题都应该回答“是”。

  1. 实现、文档和测试
    1. 实现是否易于理解（是否有有意义的变量和函数名称）？
    2. API 是否有足够的文档？
    3. 是否有足够的测试覆盖各种情况？
  2. API 设计
    1. API 是否自我文档化，以便用户可以轻松理解其使用？
    2. API 风格是否与我们现有的 API 一致？
    3. API 是否合理利用类型系统来强制执行约束？
    4. 失败路径和错误消息是否清晰？
    5. 所有破坏性更改是否有正当理由并已记录？
  3. 功能
    1. 该功能在计算机科学术语中是否合逻辑？
    2. 该功能是否确实能够与我们支持的所有后端工作？
    3. 所有注意事项是否已讨论并消除/记录？
  4. 架构
    1. 该功能是否与我们代码库的现有架构相符合？
    2. 是否不会造成技术负债/维护负担？
    3. 是否不会破坏抽象？

1、2 和 3 相对客观事实，然而对于 4 的答案可能需要一些讨论和辩论。如果无法达成共识，@tyt2y3 将做出最终裁决。

目前的审阅者是谁？​

截至目前，SeaQL 有 3 名核心成员兼审阅者：

Chris Tsang
创始人。负责维护所有项目。
Billy Chan
创始成员。SeaORM 和 Seaography 的共同维护者。
Ivan Krivosheev
2022 年加入。SeaQuery 的共同维护者。

如何成为审阅者？​

我们将邀请一些与我们紧密合作的贡献者，但您也可以自愿加入——要求是：您已对我们的项目做出了实质性代码贡献，并展现了对我们 工程实践 的熟悉。

随着时间的推移，当您对我们的组织做出了重要贡献时，您也可以成为核心成员。

让我们共同为 Rust 的未来构建 🦀​
