宣布 SeaORM 0.12

August 12, 2023 · 11 min read
SeaQL 团队
Chris Tsang

🎉 我们很高兴今天宣布 SeaORM 0.12!

我们仍然记得两年前首次向 Rust 社区 介绍 SeaORM 的时候。我们的目标是帮助开发人员在 Rust 中构建异步数据库驱动的应用程序。

今天，许多开源项目、一些初创公司 和许多其他闭源项目都在使用 SeaORM。感谢所有参与和贡献的人！

新特性 🌟​

🧭 Seaography: GraphQL 集成（预览）​

Seaography 是一个基于 SeaORM 的 GraphQL 框架。在 0.12 版本中，Seaography 集成已内置于 sea-orm 中。Seaography 允许您快速构建 GraphQL 解析器。只需几条命令，您就可以从 SeaORM 实体启动 GraphQL 服务器！

虽然 Seaography 的开发还处于早期阶段，但在原型设计和构建内部使用的管理面板时非常有用。

阅读文档以获取更多信息。

新增宏 DerivePartialModel​

#1597 现在您可以轻松执行自定义选择，仅查询所需的列。

#[derive(DerivePartialModel, FromQueryResult)]
#[sea_orm(entity = "Cake")]
struct PartialCake {
    name: String,
    #[sea_orm(
        from_expr = r#"SimpleExpr::FunctionCall(Func::upper(Expr::col((Cake, cake::Column::Name))))"#
    )]
    name_upper: String,
}

assert_eq!(
    cake::Entity::find()
        .into_partial_model::<PartialCake>()
        .into_statement(DbBackend::Sqlite)
        .to_string(),
    r#"SELECT "cake"."name", UPPER("cake"."name") AS "name_upper" FROM "cake""#
);

新增 Select::find_with_linked​

#1728, #1743 与 find_with_related 类似，您现在可以选择相关实体并合并模型。

// 考虑以下链接
pub struct BakedForCustomer;

impl Linked for BakedForCustomer {
    type FromEntity = Entity;

    type ToEntity = super::customer::Entity;

    fn link(&self) -> Vec<RelationDef> {
        vec![
            super::cakes_bakers::Relation::Baker.def().rev(),
            super::cakes_bakers::Relation::Cake.def(),
            super::lineitem::Relation::Cake.def().rev(),
            super::lineitem::Relation::Order.def(),
            super::order::Relation::Customer.def(),
        ]
    }
}

let res: Vec<(baker::Model, Vec<customer::Model>)> = Baker::find()
    .find_with_linked(baker::BakedForCustomer)
    .order_by_asc(baker::Column::Id)
    .all(db)
    .await?;

新增 DeriveValueType 派生宏，用于 自定义包装类型​

#1720 现在您可以轻松使用新类型。

#[derive(Clone, Debug, PartialEq, Eq, DeriveEntityModel)]
#[sea_orm(table_name = "custom_value_type")]
pub struct Model {
    #[sea_orm(primary_key)]
    pub id: i32,
    pub number: Integer,
    // 仅支持 Postgres
    pub str_vec: StringVec,
}

#[derive(Clone, Debug, PartialEq, Eq, DeriveValueType)]
pub struct Integer(i32);

#[derive(Clone, Debug, PartialEq, Eq, DeriveValueType)]
pub struct StringVec(pub Vec<String>);

这为您节省了以下样板代码：

impl std::convert::From<StringVec> for Value { .. }

impl TryGetable for StringVec {
    fn try_get_by<I: ColIdx>(res: &QueryResult, idx: I)
        -> Result<Self, TryGetError> { .. }
}

impl ValueType for StringVec {
    fn try_from(v: Value) -> Result<Self, ValueTypeErr> { .. }

    fn type_name() -> String { "StringVec".to_owned() }

    fn array_type() -> ArrayType { ArrayType::String }

    fn column_type() -> ColumnType { ColumnType::String(None) }
}

增强功能 🆙​

#1433 链式 AND / OR 连接条件​

为 DeriveRelation 新增了更多宏属性。

// 实体文件

#[derive(Copy, Clone, Debug, EnumIter, DeriveRelation)]
pub enum Relation {
    // 默认是 `JOIN `fruit` ON `cake`.`id` = `fruit`.`cake_id` AND `fruit`.`name` LIKE '%tropical%'`
    #[sea_orm(
        has_many = "super::fruit::Entity",
        on_condition = r#"super::fruit::Column::Name.like("%tropical%")"#
    )]
    TropicalFruit,
    // 指定 `condition_type = "any"` 来覆盖它，现在变为
    // `JOIN `fruit` ON `cake`.`id` = `fruit`.`cake_id` OR `fruit`.`name` LIKE '%tropical%'`
    #[sea_orm(
        has_many = "super::fruit::Entity",
        on_condition = r#"super::fruit::Column::Name.like("%tropical%")"#
        condition_type = "any",
    )]
    OrTropicalFruit,
}

#1508 支持具有 12 个元素的复合主键的实体​

#[derive(Clone, Debug, PartialEq, DeriveEntityModel)]
#[sea_orm(table_name = "primary_key_of_12")]
pub struct Model {
    #[sea_orm(primary_key, auto_increment = false)]
    pub id_1: String,
    ...
    #[sea_orm(primary_key, auto_increment = false)]
    pub id_12: bool,
}

#1677 新增 UpdateMany::exec_with_returning()​

let models: Vec<Model> = Entity::update_many()
    .col_expr(Column::Values, Expr::expr(..))
    .exec_with_returning(db)
    .await?;

#1511 新增 MigratorTrait::migration_table_name() 方法来配置迁移表的名称​

#[async_trait::async_trait]
impl MigratorTrait for Migrator {
    // 覆盖迁移表的名称
    fn migration_table_name() -> sea_orm::DynIden {
        Alias::new("override_migration_table_name").into_iden()
    }
    ...
}

#1707 新增 DbErr::sql_err() 方法来解析常见数据库错误​

assert!(matches!(
    cake.into_active_model().insert(db).await
        .expect_err("插入重复主键的行")
        .sql_err(),
    Some(SqlErr::UniqueConstraintViolation(_))
));

assert!(matches!(
    fk_cake.insert(db).await
        .expect_err("插入无效外键的行")
        .sql_err(),
    Some(SqlErr::ForeignKeyConstraintViolation(_))
));

#1737 介绍新的 ConnAcquireErr​

enum DbErr {
    ConnectionAcquire(ConnAcquireErr),
    ..
}

enum ConnAcquireErr {
    Timeout,
    ConnectionClosed,
}

#1627 新增 DatabaseConnection::ping()​

|db: DatabaseConnection| {
    assert!(db.ping().await.is_ok());
    db.clone().close().await;
    assert!(matches!(db.ping().await, Err(DbErr::ConnectionAcquire)));
}

#1708 新增 TryInsert 方法，在空插入时不会 panic​

// 现在，您可以这样做：
let res = Bakery::insert_many(std::iter::empty())
    .on_empty_do_nothing()
    .exec(db)
    .await;

assert!(matches!(res, Ok(TryInsertResult::Empty)));

#1712 插入冲突时什么都不做以返回 Ok​

let on = OnConflict::column(Column::Id).do_nothing().to_owned();

// 现有行为
let res = Entity::insert_many([..]).on_conflict(on).exec(db).await;
assert!(matches!(res, Err(DbErr::RecordNotInserted)));

// 新 API；现在您可以：
let res =
Entity::insert_many([..]).on_conflict(on).do_nothing().exec(db).await;
assert!(matches!(res, Ok(TryInsertResult::Conflicted)));

#1740, #1755 用 sea_orm::DeriveIden 替换 sea_query::Iden​

为了提供更一致的接口，sea-query/derive 不再由 sea-orm 启用，因此 Iden 不再作为派生宏工作（仍然是一个特性）。

// 然后：

#[derive(Iden)]
#[iden = "category"]
pub struct CategoryEnum;

#[derive(Iden)]
pub enum Tea {
    Table,
    #[iden = "AfternoonTea"]
    EverydayTea,
}

// 现在：

#[derive(DeriveIden)]
#[sea_orm(iden = "category")]
pub struct CategoryEnum;

#[derive(DeriveIden)]
pub enum Tea {
    Table,
    #[sea_orm(iden = "AfternoonTea")]
    EverydayTea,
}

新版本 列车 渡船 🚢​

这是 SeaORM 的第 12 次发布！最初，每个月发布一个主要版本。逐渐变为每 2 到 3 个月一次，现在自上次主要发布以来已经过去了 6 个月。随着我们的用户群增长，并且一些人已经在 生产中使用 SeaORM，我们理解拥有一个稳定的 API 接口和功能集的重要性。

因此，我们承诺：

  1. 严格审查破坏性变更
  2. 扩展我们的测试套件以覆盖我们库的所有特性
  3. 永远不移除功能，并仔细考虑弃用

今天，SeaORM 的架构非常扎实和稳定，在 0.12 发布时我们偿还了很多技术债务，这将使我们能够在不破坏的情况下交付新功能和增强功能。由于我们的主要依赖 SQLx 还没有达到 1.0，因此从技术上讲我们不能成为 1.0。

我们仍在快速推进，我们将始终在 SQLx 发布新版本时尽快发布新版本，以便您可以同时升级所有内容。因此，SeaORM 的下一个主要版本将在6 个月内发布，或在 SQLx 发布新版本时，以先到者为准。
