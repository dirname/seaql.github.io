SeaStreamer 0.3 中的新功能

September 6, 2023 · 7 min read
Chris Tsang
SeaQL 团队

🎉 我们很高兴地发布了 SeaStreamer 0.3.x!

文件后端​

SeaStreamer 0.3 的一个重要更新是文件后端。它实现了相同的高级 MPMC API，允许文件之间的流式传输。有多种使用场景。例如，它可以用于从 Redis/Kafka 转储数据并在本地处理，或作为存储或传输的中间文件格式。

SeaStreamer 文件格式 .ss 非常简单。它非常像 .ndjson，但属于二进制格式。文件格式的设计目标如下：

  1. 支持二进制数据而没有编码开销
  2. 高效地进行大转储的回溯/查找
  3. 流式友好性 - 文件可以被截断而不丢失完整性

让我详细解释。

首先，SeaStreamer 文件是一个容器格式。它只关注消息流和框架，而不关注有效负载。它是为了与像 Protobuf 或 BSON 这样的二进制消息格式配对而设计的。

无编码​

JSON 和 CSV 是很好的纯文本文件格式，但它们对二进制不友好。通常，为了编码二进制数据，人们会使用 base64。这会产生高昂的编码/解码开销。在二进制协议中，分隔符 通常用于表示消息边界。因此，需要字节填充来转义字节。

在 SeaStreamer 中，我们希望完全避免编码开销。有效负载应该逐字写入磁盘。因此，文件格式围绕构造消息帧和放置校验和来确保数据被正确解释。

高效查找​

基于分隔符的协议有一个优点：字节流可以随机查找，我们总是可以无障碍地读取下一个消息。

由于 SeaStreamer 不依赖分隔符，我们无法轻易地在随机查找后对齐到消息帧。我们通过在整个文件的固定位置定期放置信标来解决这个问题。例如，假设 信标间隔 为 1024，那么在第 1024 个字节、第 2048 个字节等位置会有一个信标。然后，每次我们想要查找一个随机位置时，我们就会查找最接近的 N * 1024 字节并从那里开始读取。

这些信标也充当索引：它们包含每个单独流的摘要。因此，给定一个特定的流键和序列号（或时间戳）进行搜索，SeaStreamer 可以通过读取信标快速定位消息。无论流中的消息是稀疏的还是密集的，这一点都没有关系！

流式友好性​

截断文件应该始终是安全的。将文件分割成块应该相对简单。我们应该能够判断数据是否损坏。

SeaStreamer 通过为每个消息计算校验和，以及每个流的校验和的运行校验和来实现这一目标。目前这并不是强制要求，但理论上我们可以检测到流中是否缺少任何消息。

Redis 后端​

Redis Streams 被低估了！它们具有高吞吐量和并发性，最适合在同一主机上或靠近应用程序进行非持久流处理。

障碍可能在于库支持。Redis Streams 的 API 相对低级，并且没有很多高级库来帮助编程，而 Kafka 则有多种官方编程库。

陷阱是，使用原始 Redis API 很难最大化并发。首先，你需要对 XADD 命令进行流水线处理。你还需要定时和批量处理 XACK，以免阻塞读取和计算。当然，你想将读取和写入分开到不同的线程中。

SeaStreamer 为你打破了这些障碍，提供了类似 Kafka 的 API 体验！

基准测试​

在 0.3 中，我们进行了一些优化，以提高 Redis 和文件后端的吞吐量。我们将初始基准设置为每秒 100,000 条消息，希望随着时间的推移能够进一步改进。

我们的 微基准测试 涉及一个简单的程序，生成或消费 100,000 条消息，其中每条消息的有效载荷为 256 字节。

对于 Redis，它在同一台计算机上的 Docker 中运行。在我这台不算出色的搭载第十代英特尔 i7 的笔记本电脑上，数字大概如下：

生产者​

redis    0.5s
stdio    0.5s
file     0.5s

消费者​

redis    1.0s
stdio    1.0s
file     1.1s

这实际上意味着我们在 生产 每秒 100,000 条消息的领域中游刃有余，但仅能在 1 秒内 消费 100,000 条消息。欢迎提出性能改进建议！
