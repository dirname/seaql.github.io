SeaORM 1.0-rc.x 中的新特性

April 8, 2024 · 12 min read
SeaQL 团队
Chris Tsang

本文总结了在 SeaORM 1.0-rc.x 中引入的新特性和增强功能：

  • 1.0.0-rc.1
  • 1.0.0-rc.2
  • 1.0.0-rc.3

新特性​

刷新迁移模式定义​

#2099 我们意识到 SeaORM 的迁移脚本有时显得冗长。感谢 Loco 的巧妙设计，我们刷新了模式定义的语法。

旧的迁移脚本如下所示：

#[async_trait::async_trait]
impl MigrationTrait for Migration {
    async fn up(&self, manager: &SchemaManager) -> Result<(), DbErr> {
        manager
            .create_table(
                Table::create()
                    .table(Users::Table)
                    .if_not_exists()
                    .col(
                        ColumnDef::new(Users::Id)
                            .integer()
                            .not_null()
                            .auto_increment()
                            .primary_key(),
                    )
                    .col(ColumnDef::new(Users::Pid).uuid().not_null())
                    .col(ColumnDef::new(Users::Email).string().not_null().unique_key())
                    // ...
    }
}

现在，使用新的模式帮助程序，您可以使用简化的语法定义模式！

// 记得导入 `sea_orm_migration::schema::*`
use sea_orm_migration::{prelude::*, schema::*};

#[derive(DeriveMigrationName)]
pub struct Migration;

#[async_trait::async_trait]
impl MigrationTrait for Migration {
    async fn up(&self, manager: &SchemaManager) -> Result<(), DbErr> {
        manager
            .create_table(
                Table::create()
                    .table(Users::Table)
                    .if_not_exists()
                    .col(pk_auto(Users::Id)) // 自增长主键
                    .col(uuid(Users::Pid)) // UUID 列
                    .col(string_uniq(Users::Email)) // 唯一非空字符串列
                    .col(string(Users::Password)) // 字符串列
                    .col(string(Users::ApiKey).unique_key())
                    .col(string(Users::Name))
                    .col(string_null(Users::ResetToken)) // 可空字符串列
                    .col(timestamp_null(Users::ResetSentAt)) // 可空时间戳列
                    .col(string_null(Users::EmailVerificationToken))
                    .col(timestamp_null(Users::EmailVerificationSentAt))
                    .col(timestamp_null(Users::EmailVerifiedAt))
                    .to_owned(),
            )
            .await
    }

    // ...
}

每种常用列类型都有三个变体：

  • <COLUMN_TYPE>() 辅助函数，例如 string()，定义一个非空字符串列
  • <COLUMN_TYPE>_null() 辅助函数，例如 string_null()，定义一个可空字符串列
  • <COLUMN_TYPE>_uniq() 辅助函数，例如 string_uniq()，定义一个非空且唯一的字符串列

新的模式帮助程序可以通过导入 sea_orm_migration::schema::* 来使用。迁移库完全向后兼容，因此没有必要急于迁移旧脚本。建议在新脚本中使用新的语法，SeaORM 仓库中的所有示例已更新以进行演示。对于高级用例，旧的 SeaQuery 语法仍然可以使用。

重新设计的 SQLite 类型映射​

sea-orm#2077 sea-query#735 sea-schema#117 我们重新设计了 SeaQL 生态系统中 SQLite 的类型映射，使得 SeaQuery 和 SeaSchema 现在可以相互兼容。用 SeaQuery 编写的迁移可以被 sea-orm-cli 重新识别并生成兼容的实体！换句话说，往返已完成。

数据类型将根据 SQLite 的亲和性规则映射到 SQLite 类型：

  • INTEGER: integer、tiny_integer、small_integer、big_integer 和 boolean 存储为 integer
  • REAL: float、double、decimal 和 money 存储为 real
  • BLOB: blob 和 varbinary_blob 存储为 blob
  • TEXT: 所有其他数据类型存储为 text，包括 string、char、text、json、uuid、date、time、datetime、timestamp 等。

增强功能​

  • #2137 DerivePartialModel 宏属性 entity 现在支持 syn::Type
#[derive(DerivePartialModel)]
#[sea_orm(entity = "<entity::Model as ModelTrait>::Entity")]
struct EntityNameNotAIdent {
    #[sea_orm(from_col = "foo2")]
    _foo: i32,
    #[sea_orm(from_col = "bar2")]
    _bar: String,
}
  • #2146 添加了 RelationDef::from_alias()
assert_eq!(
    cake::Entity::find()
        .join_as(
            JoinType::LeftJoin,
            cake_filling::Relation::Cake.def().rev(),
            cf.clone()
        )
        .join(
            JoinType::LeftJoin,
            cake_filling::Relation::Filling.def().from_alias(cf)
        )
        .build(DbBackend::MySql)
        .to_string(),
    [
        "SELECT `cake`.`id`, `cake`.`name` FROM `cake`",
        "LEFT JOIN `cake_filling` AS `cf` ON `cake`.`id` = `cf`.`cake_id`",
        "LEFT JOIN `filling` ON `cf`.`filling_id` = `filling`.`id`",
    ]
    .join(" ")
);
  • #1665 [sea-orm-macro] 在 DeriveActiveModel 宏中限定特性
  • #2064 [sea-orm-cli] 修复空 mod.rs 文件上的 migrate generate

破坏性变更​

  • #2145 将 ConnectOptions::pool_options() 重命名为 ConnectOptions::sqlx_pool_options()
  • #2145 将 sqlx_common 设为私有，隐藏 sqlx_error_to_xxx_err
  • MySQL 的 money 类型映射为 decimal
  • MySQL 的 blob 类型移至 extension::mysql::MySqlType；ColumnDef::blob() 现在不接受参数
assert_eq!(
    Table::create()
        .table(BinaryType::Table)
        .col(ColumnDef::new(BinaryType::BinaryLen).binary_len(32))
        .col(ColumnDef::new(BinaryType::Binary).binary())
        .col(ColumnDef::new(BinaryType::Blob).custom(MySqlType::Blob))
        .col(ColumnDef::new(BinaryType::TinyBlob).custom(MySqlType::TinyBlob))
        .col(ColumnDef::new(BinaryType::MediumBlob).custom(MySqlType::MediumBlob))
        .col(ColumnDef::new(BinaryType::LongBlob).custom(MySqlType::LongBlob))
        .to_string(MysqlQueryBuilder),
    [
        "CREATE TABLE `binary_type` (",
            "`binlen` binary(32),",
            "`bin` binary(1),",
            "`b` blob,",
            "`tb` tinyblob,",
            "`mb` mediumblob,",
            "`lb` longblob",
        ")",
    ]
    .join(" ")
);
  • ColumnDef::binary() 设置列类型为 binary，默认长度为 1
  • 移除了 BlobSize 枚举
  • 添加了 StringLen 表示 varchar / varbinary 的长度
/// var-char/binary 的长度；默认为 255
pub enum StringLen {
    /// 字符串大小
    N(u32),
    Max,
    #[default]
    None,
}
  • ValueType::columntype() 的 Vec<u8> 映射为 VarBinary(StringLen::None)
  • ValueType::columntype() 的 String 映射为 String(StringLen::None)
  • ColumnType::Bit 映射为 Postgres 的 bit
  • ColumnType::Binary 和 ColumnType::VarBinary 映射为 Postgres 的 bytea
  • Value::Decimal 和 Value::BigDecimal 映射为 SQLite 的 real
  • ColumnType::Year(Option<MySqlYear>) 变更为 ColumnType::Year

发布规划​

在 SeaORM 的上一个版本中，我们表示希望下一个发布是 1.0。我们现在确实非常接近 1.0！

虽然 0.12 在 1.0 确定之前仍将得到维护，但欢迎您今天尝试 1.0-rc.x！仍然会有一些小但在技术上是破坏性的变更：

  1. #2185 向 PrimaryKeyTrait 添加 Trait 常量 ARITY，允许用户编写更好的泛型代码
  2. #2186 将 ActiveModel 关联到 EntityTrait，允许用户扩展实体的行为

现在也是您提出对 SeaORM 有长期影响的破坏性变更的完美时机。在稳定之后，我们希望 SeaORM 能够提供一个稳定的 API 接口，以便开发人员在未来几年中在生产中使用。

我们每年不会有超过 2 次主要版本发布，每个主要版本将至少维护一年。虽然目前还不确定，但这是我们现在所考虑的。此外，这实际上将使我们能够更频繁地发布新功能！

SQL Server 支持​

我们一直在计划SeaORM 的 SQL Server支持，但在 2023 年搁置了这项工作（我感到非常遗憾）。不过，SQL Server 支持很快就会到来！它将首先作为封闭的 Beta 发布给我们的合作伙伴。如果您感兴趣，请加入我们的等候名单。
