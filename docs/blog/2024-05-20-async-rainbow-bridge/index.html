同步与异步 Rust 之间的彩虹桥

May 20, 2024 · 11 min read
Chris Tsang
SeaQL 团队

这个故事源于“你的函数是什么颜色？”的说法，作为对常见编程语言异步实现的批评。嗯，Rust 也属于“有颜色的函数”。因此，在这篇博客中，让我们看看如何设计系统以有效地结合同步和异步代码。

彩虹桥引用了《雷神》中传送你到不同领域的桥——一个完美的类比！

背景​

同步代码可以是阻塞IO或昂贵的计算。异步代码通常是网络IO，你会等待结果。

在这两种情况下，我们都希望最大化并发，以便程序能够充分利用 CPU，而不是闲置。一个常见的方法是消息传递，在这里我们打包任务并将它们发送到不同的工作线程执行。

Sync -> Sync​

首先，从经典的示例开始，纯同步代码。标准库中存在 std::sync::mpsc，让我们来看看。

use std::sync::mpsc::channel;

// 创建一个无界通道
let (sender, receiver) = channel();

// 不会阻塞
sender.send("Hello".to_string()).unwrap();

let handle = std::thread::spawn(move|| {
    // 等待直到有消息
    let message = receiver.recv().unwrap();
    println!("{message}");
});

handle.join().unwrap();
println!("Bye");

输出 (Playground)：

Hello
Bye

现在，我们将做一个更详细的示例：一个程序生成多个工作线程来执行一些“昂贵”的计算。主线程会将任务分发到那些线程，然后通过另一个通道收集结果。

┌─────────────┐    任务    ┌─────────────────┐   结果
│             ╞═════════════╡ worker thread 1 ╞═════════════╗    ┌─────────────┐ 
│ main thread │             ├─────────────────┤             ╠════╡ main thread │ 
│             ╞═════════════╡ worker thread 2 ╞═════════════╝    └─────────────┘ 
└─────────────┘             └─────────────────┘

首先，设置通道。

let (result, collector) = channel(); // 结果
let mut senders = Vec::new();
for _ in 0..THREADS {
    let (sender, receiver) = channel(); // 任务
    senders.push(sender);
    let result = result.clone();
    std::thread::spawn(move || worker(receiver, result));
}

工作线程看起来像：

fn worker(receiver: Receiver<Task>, sender: Sender<Done>) {
    while let Ok(task) = receiver.recv() {
        let result = process(task);
        sender.send(result).unwrap();
    }
}

然后，分发任务。

for c in 0..TASKS {
    let task = some_random_task();
    senders[c % THREADS].send(task).unwrap();
}

最后，我们可以收集结果。

for _ in 0..TASKS {
    let result = collector.recv().unwrap();
    println!("{result:?}");
}

Async -> Async​

接下来，我们将迁移到异步领域。使用 tokio::sync::mpsc，它与上面的示例非常相似，只是每个操作都是 async，所以对生命周期施加了额外的限制。（关键是，只需移动/克隆。不要借用）

tokio 的 unbounded_channel 相当于 std 的 channel。否则它是非常相似的。spawn 方法接收一个 Future；由于工作线程需要接收通道，因此我们用 async move {} 构造一个异步闭包。

stdtokio
(unbounded) channelunbounded_channel
sync_channel(bounded) channel
let (result, mut collector) = unbounded_channel();
let mut senders = Vec::new();
for _ in 0..WORKERS {
    let (sender, mut receiver) = unbounded_channel();
    senders.push(sender);
    let result = result.clone();
    tokio::task::spawn(async move {
        while let Some(task) = receiver.recv().await {
            result.send(process(task).await).unwrap();
        }
    });
}
std::mem::drop(result); // <-- ?

为什么我们需要丢弃 result 发送者？这是一个脚趾枪：tokio 会吞掉任务内部产生的 panic，因此如果发生这种情况，程序将永远不会退出。通过丢弃作用域内的最后一个 result 副本，通道将在所有任务退出后自动关闭，这反过来会触发我们的 collector。

其余部分几乎是相同的。

for (i, task) in tasks.iter().enumerate() {
    senders[i % WORKERS].send(task.clone()).unwrap();
}
std::mem::drop(senders);

for _ in 0..tasks.len() {
    let result = collector.recv().await.unwrap();
    println!("{result:?}");
}

完整源代码可以在 这里找到。

Flume mpmc​

mpmc - 多生产者，多消费者​

前面的示例存在一个缺陷：我们必须生成多个 mpsc 通道来发送任务，这样就：

  1. 繁琐。我们需要保持一份 senders 列表
  2. 效率不是最高的。轮询是分发任务的最佳方式吗？某些工作线程可能会保持空闲

理想的设置如下：

                      任务   ┌─────────────────┐   结果
┌─────────────┐   ╔═══════════╡ worker thread 1 ╞════════════╗   ┌─────────────┐ 
│ main thread ╞═══╣           ├─────────────────┤            ╠═══╡ main thread │ 
└─────────────┘   ╚═══════════╡ worker thread 2 ╞════════════╝   └─────────────┘ 
                              └─────────────────┘

让我们使用 Flume 重写我们的示例。但首先，了解 tokio 和 flume 之间的映射：

TokioFlume
unbounded_channelunbounded (通道)
(bounded) channelbounded (通道)
sendsend
recvrecv_async

在 tokio 中，该方法是独占的：async fn recv(&mut self)；在 flume 中，该方法是 fn recv_async(&self) -> RecvFut。类型签名已经告诉你 mpsc 与 mpmc 之间的区别！在异步上下文中使用阻塞的 recv 方法是错误的，但可悲的是，编译器并不会警告你。

通道的设置现在稍微简单一些：

let (sender, receiver) = unbounded(); // 任务
let (result, collector) = unbounded(); // 结果

for _ in 0..WORKERS {
    let receiver = receiver.clone();
    let result = result.clone();
    tokio::task::spawn(async move {
        while let Ok(task) = receiver.recv_async().await {
            result.send(process(task).await).unwrap();
        }
    });
}

我们不再需要自己分发任务。所有工作线程共享同一个任务队列，因此工作线程在完成前一个任务后会立即获取下一个任务——有效地进行负载均衡！

for task in &tasks {
    sender.send(task.clone()).unwrap();
}

for _ in 0..tasks.len() {
    let result = collector.recv_async().await.unwrap();
    println!("{result:?}");
}

Sync -> Async​

在最后一个示例中，我们考虑一个大部分是同步的程序，但有一些我们希望在后台线程中处理的异步操作。

在下面的示例中，我们的阻塞操作是从主线程“读取标准输入”。我们将这些行发送到单独的异步线程进行处理。

┌─────────────┐           ┌──────────────┐
│ main thread ╞═══════════╡ async thread │
└─────────────┘           └──────────────┘

它遵循通常的 3 个步骤：

  1. 创建一个 flume 通道
  2. 将接收端传递给工作线程
  3. 通过通道发送任务
fn main() -> Result<()> {
    let (sender, receiver) = unbounded(); // flume 通道

    std::thread::spawn(move || {
        // 这个运行时是单线程的
        let rt = tokio::runtime::Builder::new_current_thread().enable_all().build().unwrap();
        rt.block_on(handler(receiver))
    });

    loop {
        let mut line = String::new();
        // 这会阻塞当前线程，直到有新行输入
        match std::io::stdin().read_line(&mut line) {
            Ok(0) => break, // 这意味着标准输入被关闭
            Ok(_) => (),
            Err(e) => panic!("{e:?}"),
        }
        sender.send(line)?;
    }

    Ok(())
}

这是处理程序：

async fn handler(receiver: Receiver<String>) -> Result<()> {
    while let Ok(line) = receiver.recv_async().await {
        process(line).await?;
    }
    Ok(())
}

看起来与 async -> async 示例没什么不同，唯一的区别是其中一侧是同步的！完整源代码可在 这里找到。

优雅地关闭​

上述代码有一个问题：我们不知道行是否已被处理。如果程序有一个从处理 sigint 的退出机制，就有可能在所有行都处理之前就退出。

让我们看看如何正确地关闭。

let handle = std::thread::spawn(..);

// running 是一个 AtomicBool
while running.load(Ordering::Acquire) {
    let line = read_line_from_stdin();
    sender.send(line)?;
}

std::mem::drop(sender);
handle.join().unwrap().unwrap();

关闭序列有 3 个步骤：

  1. 我们首先获得对线程的 JoinHandle
  2. 我们丢弃所有 sender 的副本，从而有效地关闭通道
  3. 在工作线程中，receiver.recv_async() 将会返回错误，如 文档 所述

    异步接收通道中的值，如果所有发送者都已丢弃，则返回错误。

  4. 工作线程完成，与主线程连接

结论​

同步异步
启动工作线程std::thread::spawntokio::task::spawn
并发多线程可以是多线程或单线程
工作线程是FnOnceFuture
发送消息使用sendsend
接收消息使用recvrecv_async
等待消息阻塞让渡给运行时
