🎉 我们很高兴地宣布今天发布 [SeaORM 1.0](https://github.com/SeaQL/sea-orm/releases/tag/1.0.0)！这是我们特别的时刻，所以这篇博文将会比一般的发布说明多一点内容。

## 我们的旅程

距离我们发布 [SeaORM 0.2](https://www.sea-ql.org/blog/2021-09-20-introducing-sea-orm/) 已经快三年了。当时，我们设立了一个目标：为异步 Rust 生态系统构建一个 SQL ORM，汇聚最佳的库，以便开发人员构建高性能和健壮的网络服务。

我们要感谢所有早期采用者、贡献者和SeaORM的赞助商。感谢所有用户对我们的信任，并成为我们旅程的一部分。

如今，许多初创公司和企业正在使用 Rust 构建应用程序，SeaORM 是其技术栈的重要组成部分。我们特别高兴看到像 [Loco](https://loco.rs/) 这样的应用框架，可以与 SeaORM 紧密集成，从而提供流畅的开发体验。

我认为我们已经实现了我们的初始目标！我们理解，对于团队在选择技术时，成熟度和稳定性是重要的考量。因此，我们相信现在是稳定 SeaORM 的时候。

## 新特性

以下是 SeaORM 1.0 引入的一些新特性和增强功能的亮点。

### 刷新的迁移模式定义

[#2099](https://github.com/SeaQL/sea-orm/pull/2099) 感谢 Loco 的巧妙设计，我们刷新了模式定义语法。

旧的迁移脚本看起来像这样：

```rust
#[async_trait::async_trait]
impl MigrationTrait for Migration {
    async fn up(&self, manager: &SchemaManager) -> Result<(), DbErr> {
        manager
            .create_table(
                Table::create()
                    .table(Users::Table)
                    .if_not_exists()
                    .col(
                        ColumnDef::new(Users::Id)
                            .integer()
                            .not_null()
                            .auto_increment()
                            .primary_key(),
                    )
                    .col(ColumnDef::new(Users::Pid).uuid().not_null())
                    .col(ColumnDef::new(Users::Email).string().not_null().unique_key())
                    // ...
    }
}

现在，使用新的模式辅助工具，您可以使用简化的语法定义模式！

// 记得导入 `sea_orm_migration::schema::*`
use sea_orm_migration::{prelude::*, schema::*};

#[derive(DeriveMigrationName)]
pub struct Migration;

#[async_trait::async_trait]
impl MigrationTrait for Migration {
    async fn up(&self, manager: &SchemaManager) -> Result<(), DbErr> {
        manager
            .create_table(
                Table::create()
                    .table(Users::Table)
                    .if_not_exists()
                    .col(pk_auto(Users::Id)) // 主键，自增
                    .col(uuid(Users::Pid)) // UUID 列
                    .col(string_uniq(Users::Email)) // 带唯一性和非空约束的字符串列
                    .col(string(Users::Password)) // 字符串列
                    .col(string(Users::ApiKey).unique_key())
                    .col(string(Users::Name))
                    .col(string_null(Users::ResetToken)) // 可空字符串列
                    .col(timestamp_null(Users::ResetSentAt)) // 可空时间戳列
                    .col(string_null(Users::EmailVerificationToken))
                    .col(timestamp_null(Users::EmailVerificationSentAt))
                    .col(timestamp_null(Users::EmailVerifiedAt))
                    .to_owned(),
            )
            .await
    }

    // ...
}

对于每种常用列类型，有三种变体可供选择：

  • <COLUMN_TYPE>() 辅助函数，例如 string()，定义一个非空字符串列
  • <COLUMN_TYPE>_null() 辅助函数，例如 string_null()，定义一个可空字符串列
  • <COLUMN_TYPE>_uniq() 辅助函数，例如 string_uniq()，定义一个非空且唯一的字符串列

新的模式辅助工具可以通过导入 sea_orm_migration::schema::* 使用。迁移库完全向后兼容，因此不必急于迁移旧脚本。我们建议新脚本使用新语法，SeaORM 存储库中的所有示例已更新以供演示。对于高级用例，仍然可以使用旧的 SeaQuery 语法。

重新设计的 SQLite 类型映射​

sea-orm#2077 sea-query#735 sea-schema#117 我们重新设计了 SeaQL 生态系统中 SQLite 的类型映射，使得 SeaQuery 和 SeaSchema 现在彼此兼容。使用 SeaQuery 写的迁移可以被 sea-orm-cli 重新发现，并生成兼容的实体！换句话说，双向转换已经完成。

数据类型将根据 SQLite 的亲和规则映射到 SQLite 类型，采用自定义命名方案：

  • INTEGER：integer、tiny_integer、small_integer、big_integer 和 boolean 存储为 integer
  • REAL：float、double、decimal 和 money 存储为 real
  • BLOB：blob 和 varbinary_blob 存储为 blob
  • TEXT：其他所有数据类型存储为 text，包括 string、char、text、json、uuid、date、time、datetime、timestamp 等。

完整的类型映射表可以在 这里 查阅。有关更多信息，请参考我们之前的 博文。

对 EntityModel 和 ActiveEnum 支持替代命名方式​

#2170 为 DeriveEntityModel 和 DeriveActiveEnum 添加 rename_all 属性

#[derive(DeriveEntityModel)]
#[sea_orm(table_name = "user", rename_all = "camelCase")]
pub struct Model {
    #[sea_orm(primary_key)]
    id: i32,
    first_name: String, // firstName
    #[sea_orm(column_name = "lAsTnAmE")]
    last_name: String, // lAsTnAmE
}

#[derive(EnumIter, DeriveActiveEnum)]
#[sea_orm(rs_type = "String", db_type = "String(StringLen::None)", rename_all = "camelCase")]
pub enum TestEnum {
    DefaultVariant, // defaultVariant
    #[sea_orm(rename = "kebab-case")]
    VariantKebabCase, // variant-kebab-case
    #[sea_orm(rename = "snake_case")]
    VariantSnakeCase, // variant_snake_case
    #[sea_orm(string_value = "CuStOmStRiNgVaLuE")]
    CustomStringValue, // CuStOmStRiNgVaLuE
}

发布计划​

SQLx 0.8 最近发布！我们希望尽快升级到 0.7。但是，由于 sea-orm 1.0 已经释放候选版本一段时间，并考虑到 sqlx 0.8 中的破坏性变化，我们决定将 sea-orm 1.0 保持在 sqlx 0.7。

我们计划很快在 sqlx 0.8 上发布 sea-orm 1.1。为了避免意外的构建失败，我们建议所有用户用 tilde 要求指定 SeaORM 依赖：

sea-orm = { version = "~1.0" }

根据 Cargo Book，这将防止自动升级到 1.1，因此您可以在方便的时候进行升级。

如果您依赖于 sea-orm 0.12，我们建议您今天升级到 1.0。由于它们都使用 sqlx 0.7，此次升级聚焦于 SeaORM。然后升级到 1.1，后者专注于 SQLx，因为我们不会引入任何破坏性变化。我们建议您逐步进行升级！

SeaORM 1.x 将至少维护一年。在那时，我们将决定是否发布 2.0 和/或延长 1.x 的生命周期。

我们的未来​

您可能会问，这是否意味着 SeaORM "完成" 了？不，绝对不是！SeaORM 1.0 为我们和社区提供了一个坚实的基础，可以围绕 SeaORM 构建更雄心勃勃的功能。以下是我们愿景的一些方面：

  1. 通过 Seaography 支持 GraphQL。我们希望进一步开发 Seaography，让开发人员将一组 SeaORM 实体转换为一个完整的 GraphQL 服务器！

  2. 管理仪表板。我们希望为 SeaORM 构建一个一流的前端框架。使用 SeaORM 开发管理/客户端门户将变得轻而易举！

  3. 数据科学/分析。SeaORM 现在主要关注 OLTP，我们知道人们也在使用 SeaORM 处理 OLAP 工作负载。我们希望提供更好的与数据框架库的集成，并开发更多针对数据科学和工程的功能。

  4. 扩展功能。我们希望开发帮助应用程序扩展的功能，例如分片、缓存和多租户。

SQL Server 支持​

我们已经计划了相当长一段时间的 SQL Server for SeaORM，SQL Server 终于来到了 SeaORM 1.0！它将首先作为封闭测试版提供给我们的合作伙伴。如果您感兴趣，请加入我们的 等待名单。
```
