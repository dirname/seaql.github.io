稳定与正确

测试是 Rust 编程中不可或缺的一部分。你会发现，cargo test 是内置的。

如果你不使用 unsafe 并且代码能够编译，那么你的 Rust 程序就是 安全 的。然而，它并不会自动变得 稳定。如果你在错误处理上不够小心，你的程序仍然会出现意外的 panic。

就算你的程序没有 panic，也并不意味着它是 正确 的。它仍然可能表现异常并导致数据混乱。

通过编写适当的测试，你可以提高程序的正确性。

错误类型​

首先，我们来分类数据驱动应用中的不同错误原因：

1. 类型错误​

  1. 拼写错误或不存在的符号（表或列）名称
  2. 对数据使用不兼容的函数或运算符（例如，两个字符串相加）
  3. 无效的 SQL 查询
    • 例如，JOIN 查询中的模糊符号
    • 例如，忘记在非空列上插入数据
    • 例如，在 GROUP BY 查询中忘记对每一列进行聚合

2. 事务错误​

  1. 未能维护实体关系
  2. 未能维持数据一致性和约束

3. 行为错误​

  1. 在错误条件下进行联接或过滤
  2. 不完整或不正确的查询结果
  3. 插入、更新或删除操作的隐性副作用
  4. 任何其他不按预期的行为

关于“隐性副作用”的说明：除非关系为严格的父子关系，否则不要使用 CASCADE

缓解措施​

现在，让我们看看如何缓解这些错误：

1. 类型错误​

使用 Rust 自动防止你拼写符号错误。

使用 完全静态 的查询构建器（如 Diesel）可以消除这一类错误。然而，它要求每个参数在编译时都被静态定义并可用。这是一个 苛刻 的要求，因为总有一些事情在你程序启动（环境变量）和运行（运行时配置更改）之前你无法知道。特别是如果你来自于脚本语言背景，类型系统通常是动态的，这会使这一点变得特别尴尬。

因此，SeaORM 不试图在编译时进行检查。我们计划（仍在开发中）提供对动态生成查询的运行时静态检查，你可以在单元测试中启用，但在生产中禁用。

2. 事务错误​

这些问题无法消除。通常表示你的代码存在一些逻辑错误。当它们发生时，已经为时已晚，你唯一的选择是中止。相反，它们必须主动预防：在尝试数据操作之前检查约束条件。

你应该编写一系列单元测试，以拒绝不良数据并防止其进入数据库。你的单元测试还应该验证每个 事务（在你的应用领域中，不一定是数据库事务）是否合理。

SeaORM 帮助你使用 Mock 数据库接口编写这些单元测试。

3. 行为错误​

这基本上是在领域层面测试整个程序，需要你提供种子数据并模拟常见用户操作。通常，你会在 CI 中针对真实数据库进行测试。然而，SeaORM 鼓励你缩小这些测试，以便通过 Cargo 的 集成测试 测试最重要的数据流。

由于 SeaORM 通过 MySQL、PostgreSQL 和 SQLite 抽象，你可以使用 SQLite 作为后端测试程序的行为。它足够轻量，可以频繁、本地及在 CI 上运行。需要注意的是，SQLite 缺乏 MySQL 或 PostgreSQL 的一些高级功能，因此根据你对数据库特定功能的使用，有些逻辑无法在 SQLite 内部测试。

我们正在寻找可以模拟 MySQL 和 PostgreSQL 更高级功能的 SQLite 替代品。
