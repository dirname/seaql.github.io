比较与 Diesel

SeaORM 和 Diesel 目标相同：为您提供与数据库交互的完整解决方案。

SeaORM 和 Diesel 都支持 MySQL、PostgreSQL 和 SQLite，因此您在选择时并不受限于其中之一。虽然 Diesel 允许第三方实现自定义后端，但 SeaORM 则不支持。

我们还选择了其他不同的做法。

架构​

首先，也许是最多请求的特性，即支持异步 Rust。虽然使用异步在今天可能不会为您提供更好的性能，但在早期阶段做出异步编程的架构决策是必要的。选择 SeaORM，我们共同期待 Rust 的异步生态系统成熟。

在底层，SeaORM 与 SQLx 一起为您提供一个纯 Rust 的技术栈。Diesel 默认使用本机驱动，您可能需要一些精力去替换为纯 Rust 驱动。每个方面都有其优缺点，因此这取决于您的偏好。

SeaORM 拥有模块化设计。如果您不喜欢 ORM 的概念，您仍然可以使用 SeaQuery，这是底层查询构建器。它轻量且可以轻松集成到任何项目中。在使用 SeaORM 时，您也可以使用 SeaQuery API，因此您在享受高级抽象带来的好处的同时，依然可以在需要时利用灵活的查询构建器的强大功能。

编程范式​

除了同步与异步的基础外，Diesel 和 SeaORM 的最大区别是静态与动态。

Diesel 提供一个完全在编译时的 API，类型可以完全静态检查。您也可以在 Diesel 中进行动态查询，但这将失去一些编译时类型检查的好处。

SeaORM 是动态的，事物在运行时建立。它提供了更多的灵活性。虽然您失去了一些编译时检查，但 SeaORM 通过测试帮助您证明正确性。

这两个库大量使用特征和泛型，但 SeaORM 生成的类型较少：Diesel 中的每一列都是一个结构体（每个结构体实现多个特征），而 SeaORM 中的每一列都是一个枚举变体（一个实体的所有列共同形成一个枚举，实现一些特征）。

SeaORM 也没有深度嵌套的泛型（例如 A<B<C<D<E>>>>）。

架构生成器​

在 Diesel 生态系统中，有像 barrel 的优秀库，而 SeaQL 维护了用于架构构建的工具（SeaQuery）和管理的工具 (SeaSchema)。这意味着一个熟悉的 API 和统一的体验。

相似之处​

Diesel 和 SeaORM 都是以架构为中心，这意味着一切都从您现有的数据库架构开始，而不是从您的 OOP 类开始。

Diesel 和 SeaORM 都是关系型的，这意味着您可以通过定义的关系进行复杂的联接。

最后的话​

Diesel 是 Rust 生态系统中的一个成熟库。SeaORM 非常新。两者都无法替代对方。我们希望 Rust 社区能够共同成长壮大。
