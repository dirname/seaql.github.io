与 Diesel 比较

SeaORM 和 Diesel 的目标相同：为您提供与数据库接口的完整解决方案。

SeaORM 和 Diesel 都支持 MySQL、PostgreSQL 和 SQLite，因此您在选择时没有限制。虽然 Diesel 允许第三方实现自定义后端，但 SeaORM 不支持这一点。

我们选择了一些不同的方法。

架构​

首先，也许是用户请求最多的功能，异步 Rust 支持。虽然使用异步可能不会在今天为您提供更好的性能，但在早期做出异步编程的架构决策是必要的。选择 SeaORM 使我们期待 Rust 的异步生态系统能够成熟。

在底层，SeaORM 与 SQLx 一起提供纯 Rust 技术堆栈。Diesel 默认使用本地驱动程序，您可能需要付出一些努力才能将其替换为纯 Rust 驱动程序。每一方都有其优缺点，因此取决于您的偏好。

SeaORM 具有模块化设计。如果您不喜欢 ORM 的概念，您仍然会想使用 SeaQuery，这是底层的查询构建器。它轻量，并且可以轻松集成到任何项目中。使用 SeaORM 时，SeaQuery API 也可以使用，因此您在享受高级抽象的好处的同时，当需要时仍然可以拥有灵活的查询构建器的强大功能。

编程范式​

除了同步与异步的基础之外，Diesel 和 SeaORM 之间最大的区别是静态与动态。

Diesel 提供了一个在编译时能够完全检查类型的 API。虽然您也可以使用 Diesel 进行动态查询，但您将失去一些编译时类型检查的好处。

SeaORM 是动态的，事物在运行时建立。它提供了更多的灵活性。虽然您失去了一些编译时检查，SeaORM 通过测试帮助您证明正确性。

这两个库都大量使用特征和泛型，但 SeaORM 生成的类型较少：Diesel 中的每一列都是一个结构体（每个结构体实现多个特征），而 SeaORM 中的每一列是一个枚举变体（一个实体的所有列一起形成一个枚举，该枚举实现一些特征）。

SeaORM 中没有深度嵌套的泛型（例如：A<B<C<D<E>>>>）。

架构构建器​

虽然在 Diesel 生态系统中有很棒的库，例如 barrel，SeaQL 维护了架构构建（SeaQuery）和管理的工具（SeaSchema）。这意味着您可以享受熟悉的 API 和统一的体验。

相似之处​

Diesel 和 SeaORM 都是以架构为首的，这意味着一切都从您现有的数据库架构开始，而不是从您的面向对象类开始。

Diesel 和 SeaORM 都是关系型的，这意味着您可以使用定义的关系做复杂的连接。

最后的话​

Diesel 是 Rust 生态系统中一个成熟的库。SeaORM 非常新。两者不能相互替代。我们希望 Rust 社区能够共同成长。
