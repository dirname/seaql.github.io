# 消费者

`Consumer` 特性 [定义了] 流消费者的一般接口。

`KafkaConsumer` [提供了] 更多用于提交偏移量的功能。`StdioConsumer` 目前没有特定功能。

## `ConsumerOptions`

### `ConsumerMode`

共有三种模式：

#### `RealTime`

这是“原汁原味”的流消费者。它不会自动提交，因此只从现在开始消费消息。

#### `Resumable`

当进程重启时，它将从上次提交的序列继续流。

:::info
#### Kafka 语义

它将使用对这台主机唯一的一个组 ID：在物理机器上，它将使用 MAC 地址。
在 Docker 容器内部，它将使用容器 ID。
:::

#### `LoadBalanced`

你应该手动分配一个消费者组。负载均衡机制取决于具体实现。

### `ConsumerGroup`

消费者组是一个字符串，客户端用它向流服务器标识自己。这样，当你重新连接时，可以从服务器下载状态。从代理的角度来看，这就是全部。客户端可以从任何主机或网络连接。

多个消费者可以共享同一个消费者组，并同时保持与服务器的连接。通常，目的是为了实现负载均衡。确切的语义取决于后端。

:::info
#### Kafka 语义

如果多个消费者共享同一个组，那么只有该组中的一个消费者会收到一条消息，即它是负载均衡的。

但是，正是负载均衡机制让 Kafka 与众不同：

每个流被分成多个片段（称为分区），并且每个分区只会被分配给组中的一个消费者。

假设存在 2 个消费者（在组中）和 2 个分区，那么每个消费者将从一个分区接收消息，从而实现了负载均衡。

如果有 2 个消费者和 3 个分区，那么一个消费者将被分配 2 个分区，而另一个只被分配 1 个。

然而，如果流只有一个分区，即使有很多消费者，这些消息也只会被分配的消费者接收，其他消费者将处于待机模式，从而形成热故障切换设置。
:::

:::info
#### Stdio 语义

如果多个消费者共享同一个组，那么只有组中的一个会接收消息。
这是以轮询方式负载均衡的。
:::

## `next`

轮询并接收一条消息：它等待直到有新消息。

## `stream`

返回一个异步流。你不能从同一个消费者创建多个流，也不能在流式传输时执行任何操作。

它允许你做一些整洁的事情：

```rust
let items = consumer
    .stream()
    .take(num)
    .map(process_message)
    .collect::<Vec<_>>()
    .await
```

## `assign`

将此消费者分配到特定的片段。

它仅在下一个 `Consumer::seek` 或 `Consumer::rewind` 生效。

:::info
#### Kafka 语义

总是成功。这个操作是累加的。你可以将一个消费者分配到多个片段（即分区）。还有一个 `KafkaConsumer::unassign` 方法。
:::

:::info
#### Stdio 语义

反正只有一个片段 ZERO。
:::

## `rewind`

将流回退到特定的序列号。

如果消费者尚未分配，则使用片段 ZERO。

:::info
#### Kafka 语义

注意：这将重置所有已分配分区上的所有流。
:::

:::caution
#### Stdio 语义

Stdio 后端未实现此功能。
:::

## `seek`

将所有流定位到给定的时间点。它将从时间戳大于 `to` 的最早消息开始消费。

如果消费者尚未分配，则使用片段 ZERO。

:::info
#### Kafka 语义

此异步方法不安全取消。你必须等待这个未来，并且在完成之前，此 Consumer 将无法进行任何操作。
:::

:::caution
#### Stdio 语义

Stdio 后端未实现此功能。
:::
