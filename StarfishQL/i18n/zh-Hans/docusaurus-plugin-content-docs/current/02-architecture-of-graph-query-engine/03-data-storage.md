# 数据存储

## 实体的存储

每个实体的定义会导致创建一个节点表。

例如，如上一节中定义的 `crate` 实体会创建包含以下内容的表：

- 一个 `name` 主键列，用于存储 *唯一* 的 crate 标识符（例如 `serde`、`rand`、`sea-orm`）
- 一个 `attr_version` 列，将存储 crate 的 `version` 属性[^1]
  - 定义的每个属性都有自己的列，前缀为 "attr_" 以避免混淆。也支持名为 `name` 的属性，尽管不推荐，因为它将被存储为 `attr_name`。
- 一些列来存储关于 `depends` 关系的 crates 连通性[^2]。[^3]

## 关系的存储

同样，每个关系的定义会导致创建一个边表，因为 StarfishQL 中的所有关系都是多对多的。

例如，如上一节中定义的 `depends` 关系会创建包含以下内容的表：

- 一个 `from_node` 外键列，将存储 crate 的 `name`
- 一个 `to_node` 外键列，将存储 crate 的 `name`

举例来说，边缘 "*sea-orm 依赖于 serde*" 将作为一条 **记录存储，其中 `from_node = sea-orm` 和 `to_node = serde`**。为了存储这样的边缘，首先必须存在并且继续存在名称分别为 `sea-orm` 和 `serde` 的两个 crate。[^4]

[^1]: 所有属性都是 *可空的*。如果在插入节点时未指定属性，则它变为 null。

[^2]: 连通性是一组指标，用于描述对于任何给定节点，它与多少其他节点相关。更多关于此，请参阅 [计算节点连通性](02-architecture-of-graph-query-engine/05-calculating-node-connectivity.md)。

[^3]: 即使定义新关系会导致新列的创建，但在这一过程中，涉及的表中的现有数据保持不变。

[^4]: 在插入边缘之前，其两端的节点都必须已经插入。当删除节点时，所有一端是已删除节点的边缘也会被删除（*级联删除*）。
